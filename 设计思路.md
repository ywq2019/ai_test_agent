# AI 测试工具平台 — 设计思路

***

## 一、核心出发点：让测试人员少写代码

传统测试平台的问题在于**门槛高**——测试工程师需要掌握编程、维护脚本框架、理解底层驱动。\
这个项目的起点是反向思考：**能让 AI 做的事，人不做**。

| 环节    | 传统做法         | 本平台做法                 |
| ----- | ------------ | --------------------- |
| 用例设计  | 人工逐条编写       | 上传文档 → AI 分段生成        |
| 脚本编写  | Python/JS 脚本 | `{{fn(arg)}}` 占位符，零代码 |
| 报告分析  | 人工逐条看日志      | 执行完自动调 LLM 输出修复建议     |
| 覆盖度评估 | 凭经验判断        | AI 分析盲区，自动追加补充用例      |

这个思路决定了平台的**功能边界**：平台不追求取代测试工程师的判断力，而是消除重复性劳动，把人的精力集中在测试策略和质量决策上。

***

## 二、两套引擎并行，职责分离

平台内部存在两套完全独立的执行引擎：

```
┌─────────────────────────┐   ┌──────────────────────────────┐
│      UI 自动化引擎       │   │       接口自动化引擎          │
│                         │   │                              │
│  Playwright 驱动        │   │  httpx 异步驱动              │
│  页面元素感知            │   │  断言 + 变量提取 + 压测       │
│  任务 → 用例 → 执行      │   │  项目 → 用例 → 计划 → 执行   │
└─────────────────────────┘   └──────────────────────────────┘
              │                              │
              └──────────┬───────────────────┘
                         ▼
            共用基础设施层（WebSocket / DB / AI / 日志）
```

**为什么要分离而不是统一？**

两类测试的执行模型有本质差异：

- UI 测试依赖浏览器状态，天然串行，截图、DOM 操作耦合强
- 接口测试是纯 HTTP，无状态，天然适合并发和链路编排

强行统一会让两套逻辑相互污染。分离后各自演进，UI 出问题不拖累接口测试，代码也更清晰。

## 三、AI 集成统一化：一套接口，任意模型

原设计中 AI 调用分两条路径（CLI 子进程 vs 直调 API），迭代过程中发现 CLI 子进程方式存在明显局限：

- 强依赖本机安装 Claude CLI，无法部署到服务器
- 无法通过大模型配置页切换模型
- Windows 路径和编码问题偶发故障

**改造方案**：将全部用例生成（`ai_case_generator.py` / `case_generator.py` / `api_case_generator.py`）统一改为 HTTP API 调用，自动根据模型类型选择请求格式：

```python
is_anthropic = "anthropic.com" in base_url or model.startswith("claude")

if is_anthropic:
    # Anthropic Messages API
    url     = f"{base_url}/v1/messages"
    headers = {"x-api-key": api_key, "anthropic-version": "2023-06-01"}
    payload = {"system": system_prompt, "messages": [...]}
else:
    # OpenAI Chat Completions 兼容格式
    url     = f"{base_url}/v1/chat/completions"
    headers = {"Authorization": f"Bearer {api_key}"}
    payload = {"messages": [{"role": "system", ...}, {"role": "user", ...}]}
```

**改造后的架构**：

```
┌──────────────────────────────────────────────────────┐
│              大模型配置页（统一入口）                  │
│   API Key / API URL / 模型名  →  settings.AI_*       │
└───────────────────┬──────────────────────────────────┘
                    │ 所有 AI 功能统一读取
        ┌───────────┼───────────────┐
        ▼           ▼               ▼
   用例生成      接口用例生成      报告分析 / LangGraph
```

**好处**：
- 大模型配置页切换模型，所有 AI 功能同步生效
- 不依赖本机 CLI，可部署到任意服务器
- Claude / DeepSeek / GPT / Ollama 等任意模型均可使用

### 路径 B：LangGraph 对话代理（保留）

```python
用户指令 → LangGraph 状态机 → 工具调用（parse_page / generate_cases / execute_tests）→ 输出结果
```

**适用场景**：通过自然语言指令驱动完整测试流程，同样读取大模型配置。

## 四、变量池设计：打通执行链路

接口测试的核心难题是**用例间数据依赖**——登录接口拿到 token，后续接口必须带上这个 token。

平台设计了三层变量机制：

```
┌────────────────────────────────────────────────┐
│  全局变量池（global）                           │
│  跨项目、持久化到 DB                           │
│  引用方式：{{gvar:token}}                      │
├────────────────────────────────────────────────┤
│  本地变量（local）                              │
│  当次执行链内共享，内存级                       │
│  引用方式：{{var:user_id}}                     │
├────────────────────────────────────────────────┤
│  占位符函数                                    │
│  动态计算，执行时解析                          │
│  {{uuid()}}  {{timestamp()}}  {{fn(arg1,arg2)}}│
└────────────────────────────────────────────────┘
```

**测试计划中的关键设计**：所有步骤共享同一个 `var_store` 字典，前一步通过 JSONPath 提取的变量，后一步请求体或断言中可以直接引用，实现真正的**端到端链路测试**，而不是一堆孤立的接口调用。

**自定义脚本函数**的设计也值得一提：函数以 Python 代码存入数据库，执行时动态编译注入，用 `{{generate_req_sign(app_secret, uinfo, password)}}` 这种形式调用。测试人员只需填写参数，不接触 Python 运行时。

## 五、网络层可配置：面向真实测试环境

测试环境往往不是标准公网直连，常见问题：

- 内网服务需要走代理才能访问
- 测试域名需要解析到特定 IP（DNS 隔离）
- 不同环境用不同的网络出口

平台在 HTTP 执行层做了两层配置：

### 代理（Proxy）

```
支持格式：
  192.168.1.100:8080          ← 自动补 http://
  http://proxy.corp.com:3128
  socks5://user:pass@host:1080
```

### Hosts 映射

```
格式同 /etc/hosts：
  47.94.236.243 japi.hqwx.com
  192.168.1.10  api.dev.example.com
```

**实现原理**：自定义 `httpx.AsyncHTTPTransport` 子类，拦截 `handle_async_request`，将目标域名的连接重定向到指定 IP，同时保留原域名作为 `Host` 请求头，HTTPS 配合 `verify=False` 正常工作。

**为什么不直接改系统 hosts 文件？**

改系统 hosts 文件会影响本机所有进程，且需要管理员权限，每次切换环境都要手动改。平台的方案做到**应用内隔离**，不同项目可以有不同的映射规则，互不干扰。

**优先级链路**：

```
测试计划级 hosts > 项目级 hosts（同名条目计划覆盖，不冲突条目合并）
测试计划级代理  > 项目级代理
```

## 六、实时推送：全程 WebSocket，不轮询

所有长时间运行的任务（用例生成、测试执行、压力测试、AI 分析）均通过 WebSocket 推送进度，前端**不做任何轮询**。

### 频道设计（按 client\_id 分组广播）

| client\_id  | 事件                     | 场景                  |
| ----------- | ---------------------- | ------------------- |
| `cases_gen` | `cases_gen_progress`   | UI 用例生成进度           |
| `api_exec`  | `api_exec_progress`    | 接口用例执行结果            |
| `api_load`  | `load_metrics`         | 压测 TPS/P95/P99 实时指标 |
| `plan_<id>` | `plan_step_start/done` | 测试计划步骤进度            |

不同页面订阅不同 client\_id，互不干扰。

### 后端异步模型

```python
# HTTP 请求立即返回，执行在后台进行
background_tasks.add_task(_execute_plan_bg, plan_id)
return {"message": "已开始执行", "plan_id": plan_id}

# 后台任务通过 WS 推送每步结果
await ws_manager.broadcast_all({"type": "plan_step_done", ...})
```

**为什么不用轮询？**

压测场景下每秒推送一次 TPS 指标，轮询会产生大量无效请求；用例执行结果需要逐条实时展示，轮询的延迟和服务器负担都不可接受。WebSocket 是这类场景的唯一合理选择。

## 七、插件化技能架构：可扩展性

```
skills/
  ├── test_case_pro_max/      ← 放进去就自动注册
  │   ├── SKILL.md            ← 描述、配置、提示词
  │   └── scripts/run.py      ← 执行入口
  ├── page_parser/
  ├── document_parser/
  └── ...
```

**设计原则**：新增测试能力不修改主代码，扫描目录自动发现注册。

**为什么要插件化？**

测试场景千变万化，平台不可能内置所有能力。比如：

- 某项目需要特殊的签名算法 → 写一个自定义脚本注入
- 某团队想用增强提示词生成更规范的用例 → 新建技能目录
- 未来要支持 gRPC 测试 → 新建 grpc\_executor 技能

插件架构让平台的能力边界可以被业务方自己扩展，不依赖平台维护者。

## 八、数据库设计：向前兼容，零停机升级

使用 SQLite + SQLAlchemy ORM，启动时自动建表，新字段通过 `ALTER TABLE` 兼容旧库：

```python
for ddl in [
    "ALTER TABLE api_projects ADD COLUMN proxy_url VARCHAR(512) DEFAULT ''",
    "ALTER TABLE api_projects ADD COLUMN hosts_map TEXT DEFAULT ''",
    "ALTER TABLE test_plans ADD COLUMN proxy_url VARCHAR(512) DEFAULT ''",
]:
    try:
        await conn.execute(text(ddl))
    except Exception:
        pass  # 列已存在则忽略，不报错
```

**为什么这样设计？**

面向个人/小团队的工具，不能要求用户在升级前先备份数据库、执行迁移脚本。\
这个方案的升级流程是：**直接拉代码 → 重启服务 → 旧数据库自动补列，历史数据完整保留**。

代价是不支持列的重命名和删除，但对于快速迭代的工具产品，这是合理的取舍。

## 九、前端设计原则

### 统一的实时体验

所有耗时操作都有进度反馈，没有「点完按钮然后盯着空白页等」的场景：

- 进度弹窗 + 百分比（用例生成/优化）
- 步骤列表实时滚动（测试计划执行）
- 实时折线图（压测指标）

### 数据展示层级化

报告不是一张大表格，而是分层展示：

- **卡片列表**：快速扫描整体通过率
- **详情抽屉**：展开看步骤级断言结果
- **AI 分析面板**：结构化解析，按标题/要点/正文分块渲染，支持复制

### 操作效率优化

- 链接式操作按钮（执行 | 编辑 | 删除）+ 分隔线，替代色块按钮，减少视觉噪声
- 批量操作工具栏（多选 + 批量删除/启用/禁用）
- 时间统一处理 UTC 偏移问题（后端存 UTC，前端补 `Z` 后缀转本地时区）
- 任务管理状态/环境列中文映射（`parsed` → 已解析、`test` → 测试环境等），原始值保留在数据层
- AI 用例生成模块内支持多选 + 批量删除，不同模块选中状态互相独立

### AI 分析报告展示优化

接口测试报告和测试计划报告的 AI 分析结果通过 `marked.js` 渲染 Markdown，做了两层优化：

**后端 prompt 格式约束**：system prompt 明确要求 `##` 一级标题、`-` 列表、禁用 `---` 分隔线，避免 DeepSeek 等模型输出杂乱格式。

**前端 CSS 精细化**：
- `h2` 加左侧蓝色竖线 + 渐变背景，章节层次清晰
- 列表符号统一蓝色，表格偶数行灰底
- 白底卡片替代灰底，视觉更干净

## 十一、多用户并发设计

最初架构是单机单用户设计，`UITestAgent`、`BrowserPool`、`TestExecutor` 都持有全局共享状态，多用户同时操作会互相覆盖。经过方案一改造后，已支持团队局域网共享使用。

### 核心改造思路

**原问题：**
```
用户 A 执行 → uitest_agent.state.cases = A的用例
用户 B 执行 → uitest_agent.state.cases = B的用例  ← 覆盖了 A！
```

**改造后：**
```
用户 A 执行 → uitest_agent._states[task_id_A].cases = A的用例
用户 B 执行 → uitest_agent._states[task_id_B].cases = B的用例  ← 完全独立
```

### 三层隔离

**1. 浏览器 Context 隔离（`tools/browser.py`）**

```python
class BrowserPool:
    # 共享 Browser 实例（进程级，避免反复启动开销）
    # 每次请求创建独立 Context（完全隔离 cookie/storage/状态）
    # Semaphore(3) 限制最大并发，超出排队等待

    async def acquire(self, browser_type) -> BrowserTool:
        await self._semaphore.acquire()   # 排队等槽位
        browser = await self._ensure_browser(browser_type)
        bt = BrowserTool(browser, browser_type)
        await bt._init_context()          # 独立 Context
        return bt

    # 用完必须调用 release()，归还槽位
```

**2. Agent 状态隔离（`agent/core.py`）**

```python
class UITestAgent:
    def __init__(self):
        self._states: Dict[int, AgentState] = {}  # 按 task_id 隔离

    def _get_state(self, task_id: int) -> AgentState:
        if task_id not in self._states:
            self._states[task_id] = AgentState(task_id)
        return self._states[task_id]

    def _cleanup_state(self, task_id: int):
        self._states.pop(task_id, None)   # 任务结束自动清理
```

**3. 执行状态隔离（`skills/test_executor.py`）**

```python
class TestExecutor:
    def __init__(self):
        self._states: Dict[int, TaskExecutionState] = {}

    # 暂停/继续/停止精确控制单个任务，不影响其他用户
    def pause_task(self, task_id: int): ...
    def resume_task(self, task_id: int): ...
    def stop_task(self, task_id: int): ...
```

### 部署方式

**本机开发（Windows）**：直接 `python main.py`，使用 SQLite，零配置。

**服务器 Docker 部署**：`docker compose up -d`，自动启动 PostgreSQL + 应用，支持多人并发。

```
docker compose up -d
       ↓
  ┌─────────────┐     ┌──────────────────┐
  │  app 容器   │────▶│  db 容器         │
  │  FastAPI    │     │  PostgreSQL 15   │
  │  4000 端口  │     │  自动建表        │
  └─────────────┘     └──────────────────┘
       ↓
  Volume 持久化（数据库 / 报告 / 截图 / 日志）
```

两套环境数据库不同，代码完全一致，SQLAlchemy 自动兼容。

| 环境 | 数据库 | 适合人数 |
| --- | --- | --- |
| 本机开发 | SQLite | 1-5 人 |
| 服务器 Docker | PostgreSQL | 不限 |

## 十二、一句话总结

> 平台的设计主线是**把 AI 能力嵌入测试生命周期的每个环节**，用插件化架构保证可扩展，用 WebSocket 保证实时体验，用可配置的网络层适配真实测试环境，用共享变量池打通接口链路，用统一的 HTTP API 适配任意大模型，用三层状态隔离支持团队并发使用，整体目标是让测试工程师从重复劳动中解放出来，专注于测试策略和质量判断。

## 附：技术选型对照表

| 问题         | 选型                   | 原因                                 |
| ---------- | -------------------- | ---------------------------------- |
| 异步 HTTP 服务 | FastAPI + Uvicorn    | 原生 async/await，性能好，自动生成 API 文档     |
| 数据库        | SQLite（本机）/ PostgreSQL（Docker） | 零配置适合开发，pg 适合生产多并发；ORM 层无缝切换 |
| 浏览器自动化     | Playwright           | 支持多浏览器，async API，比 Selenium 更稳定    |
| 接口执行       | httpx                | 原生异步，支持自定义 Transport（Hosts 映射核心依赖） |
| 实时通信       | WebSocket            | 服务端主动推送，适合进度/指标场景，拒绝轮询             |
| 前端框架       | Vue 3 + Element Plus | 组件成熟，上手快，适合工具类产品                   |
| AI 调用      | 统一 HTTP API（自动判断 Anthropic / OpenAI 格式） | 支持任意模型，大模型配置页一键切换，不依赖本机 CLI |
| 默认 AI 模型   | DeepSeek V4 Flash    | 速度快、成本低、效果好，OpenAI 兼容格式            |
| 状态管理       | Pinia                | 比 Vuex 更轻，适合中等规模应用                 |
| Markdown 渲染 | marked.js + 自定义 CSS | AI 分析结果结构化展示，层次清晰                  |
