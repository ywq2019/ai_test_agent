# AI 测试工具平台 — 设计思路

***

## 一、核心出发点：让测试人员少写代码

传统测试平台的问题在于**门槛高**——测试工程师需要掌握编程、维护脚本框架、理解底层驱动。\
这个项目的起点是反向思考：**能让 AI 做的事，人不做**。

| 环节    | 传统做法         | 本平台做法                 |
| ----- | ------------ | --------------------- |
| 用例设计  | 人工逐条编写       | 上传文档 → AI 分段生成        |
| 需求变更  | 人工逐条比对、手动改用例 | 上传新文档 → AI Diff + 用例级增量更新 |
| 脚本编写  | Python/JS 脚本 | `{{fn(arg)}}` 占位符，零代码 |
| 报告分析  | 人工逐条看日志      | 执行完自动调 LLM 输出修复建议     |
| 覆盖度评估 | 凭经验判断        | AI 分析盲区，自动追加补充用例      |

这个思路决定了平台的**功能边界**：平台不追求取代测试工程师的判断力，而是消除重复性劳动，把人的精力集中在测试策略和质量决策上。

***

## 二、两套引擎并行，职责分离

平台内部存在两套完全独立的执行引擎：

```
┌─────────────────────────┐   ┌──────────────────────────────┐
│      UI 自动化引擎       │   │       接口自动化引擎          │
│                         │   │                              │
│  Playwright 驱动        │   │  httpx 异步驱动              │
│  页面元素感知            │   │  断言 + 变量提取 + 压测       │
│  任务 → 用例 → 执行      │   │  项目 → 用例 → 计划 → 执行   │
└─────────────────────────┘   └──────────────────────────────┘
              │                              │
              └──────────┬───────────────────┘
                         ▼
            共用基础设施层（WebSocket / DB / AI / 日志）
```

**为什么要分离而不是统一？**

两类测试的执行模型有本质差异：

- UI 测试依赖浏览器状态，天然串行，截图、DOM 操作耦合强
- 接口测试是纯 HTTP，无状态，天然适合并发和链路编排

强行统一会让两套逻辑相互污染。分离后各自演进，UI 出问题不拖累接口测试，代码也更清晰。

## 三、AI 集成统一化：一套接口，任意模型

原设计中 AI 调用分两条路径（CLI 子进程 vs 直调 API），迭代过程中发现 CLI 子进程方式存在明显局限：

- 强依赖本机安装 Claude CLI，无法部署到服务器
- 无法通过大模型配置页切换模型
- Windows 路径和编码问题偶发故障

**改造方案**：将全部用例生成（`ai_case_generator.py` / `case_generator.py` / `api_case_generator.py`）统一改为 HTTP API 调用，自动根据模型类型选择请求格式：

```python
is_anthropic = "anthropic.com" in base_url or model.startswith("claude")

if is_anthropic:
    # Anthropic Messages API
    url     = f"{base_url}/v1/messages"
    headers = {"x-api-key": api_key, "anthropic-version": "2023-06-01"}
    payload = {"system": system_prompt, "messages": [...]}
else:
    # OpenAI Chat Completions 兼容格式
    url     = f"{base_url}/v1/chat/completions"
    headers = {"Authorization": f"Bearer {api_key}"}
    payload = {"messages": [{"role": "system", ...}, {"role": "user", ...}]}
```

**改造后的架构**：

```
┌──────────────────────────────────────────────────────┐
│              大模型配置页（统一入口）                  │
│   API Key / API URL / 模型名  →  settings.AI_*       │
└───────────────────┬──────────────────────────────────┘
                    │ 所有 AI 功能统一读取
        ┌───────────┼───────────────┐
        ▼           ▼               ▼
   用例生成      接口用例生成      报告分析 / LangGraph
```

**好处**：
- 大模型配置页切换模型，所有 AI 功能同步生效
- 不依赖本机 CLI，可部署到任意服务器
- Claude / DeepSeek / GPT / Ollama 等任意模型均可使用

### 路径 B：LangGraph 对话代理（保留）

```python
用户指令 → LangGraph 状态机 → 工具调用（parse_page / generate_cases / execute_tests）→ 输出结果
```

**适用场景**：通过自然语言指令驱动完整测试流程，同样读取大模型配置。

## 四、需求变更增量更新：用例级精准合并

需求变更后重新生成用例的最大问题是**旧用例全量覆盖**——已经调试过的用例、人工编辑的步骤、执行历史全部丢失。平台设计了一套三层增量更新机制：

### 第一层：AI Diff 分析（模块级）

把旧用例的真实模块名列表注入 Prompt，要求 AI 必须用这些原始名字，输出 `changed / added / removed / unchanged` 四分类：

```python
# 关键约束：防止 AI 自创模块名导致旧用例找不到
existing_module_names = [m.get("name") for m in old_cases_data["modules"]]
# 注入 Prompt，强制 AI 从列表中选取模块名
```

**兜底机制**：AI 漏分类的旧模块，用新文档文本做关键词搜索二次判断：
- 新文档搜得到 → 视为 unchanged（保留）
- 搜不到 → 视为 removed（废弃）

**通用测试模块豁免**：性能测试、兼容性测试等模块名含"性能/兼容/安全"关键词的，无论 AI 是否把它归入 removed，永远保留——需求文档不写测试规范，这些模块与需求变更无关。

### 第二层：用例级保守合并（changed 模块）

对 `changed` 模块不做整体替换，而是让 LLM 只识别「因变更导致真正失效」的用例：

```
合并策略（保守原则）：
  默认保留所有旧用例
  ↓
  LLM 只找：功能点在新文档中「完全消失」的用例 → deprecated
  LLM 识别：新需求中出现的新场景 → 生成新用例追加
  ↓
  安全兜底：deprecated 超过旧用例 50% → 自动重置为空（LLM 过激保护）
```

**为什么默认保留而不是逐条判断？**

让 LLM 逐条判断每条用例是否 keep/deprecated，当面对整个新文档时，LLM 不确定旧用例对应哪部分，倾向于"不确定就 deprecated"，导致大量旧用例被错误废弃。反向逻辑（默认保留，只找失效的）更符合测试工程师的实际操作习惯：有疑问就保留，有明确依据才废弃。

### 第三层：版本追踪

每次增量更新创建新版本记录，`parent_id` 指向旧版，旧版标记 `record_status=deprecated`，废弃用例追加在模块末尾，前端默认隐藏（可通过开关查看）：

```python
class AICaseFile:
    parent_id     # 上一版本 ID
    doc_hash      # 文档 MD5，快速判断是否变更
    doc_content   # 文档原文（截断 20000 字），供 Diff 分析
    diff_summary  # 本次变更一句话摘要
    record_status # active / deprecated
```

## 五、废弃与禁用：两个概念分开存储

早期把「需求变更废弃」和「用户手动禁用」都用 `enabled=False` 表示，产生了一个冲突：用户禁用的用例不想执行，但不代表这条用例无效（禁用可能只是暂时的）；废弃的用例则应该完全退出执行和统计。

**改造**：给 `TestCase` 加独立的 `deprecated` 字段：

| 字段 | 语义 | 谁来改 | 执行影响 | 覆盖率影响 |
|------|------|------|------|------|
| `enabled` | 用户手动启用/禁用 | 用户操作 | 由前端传 `case_ids` 控制 | 计入覆盖率 |
| `deprecated` | 需求变更废弃 | 增量更新自动写入 | **强制跳过** | **排除** |

这样禁用的用例仍然是"有效的测试设计"，只是当前不执行；废弃的用例才是真正的"已无效"。

## 六、接口测试的两个 AI 增强能力

### 接口代码驱动用例生成

传统 AI 接口用例生成依赖 Swagger 文档或自然语言描述，但测试工程师拿到的往往是开发提交的 Controller/Handler 代码，文档滞后或根本不存在。

**设计思路**：新增第三种输入模式——直接粘贴接口实现代码，AI 解析其中的：

- **入参结构**：路径参数、Query 参数、请求体字段及其类型约束
- **边界条件**：代码中显式的 `if/else`、范围判断、正则校验
- **异常路径**：`raise`、`return error`、状态码分支
- **隐式业务规则**：注释中的限制说明、数据库唯一约束等

```python
# generate_from_code() 核心逻辑
# 1. 解析代码结构 → 识别入参/边界/分支/异常路径
# 2. 生成三类优先级用例：
#    P0 — 正常主流程（基本入参，断言业务成功）
#    P1 — 参数校验/业务异常（缺少必填参数、越界值、权限不足）
#    P2 — 边界值（最大值/最小值/空字符串/特殊字符）
# 3. 每条用例附 _code_insight 字段，记录从代码识别的关键测试依据
```

支持 Python/Java/Go/Node.js/PHP 六种语言，进度通过 WebSocket 推送，与 Swagger 生成走同一条路径写入用例库。

### 代码可行性分析：需求 vs 实现的 AI 对比

代码评审关注的是"代码写得对不对"，而测试工程师更关心"代码实现了需求吗"——这两个问题用不同工具解决。

**设计思路**：同时提供需求文档和接口实现代码，AI 做双向对比，识别四类偏差：

| 偏差类型 | 含义 | 典型例子 |
| --- | --- | --- |
| `missing` | 需求有但代码未实现 | 需求要求返回分页总数，代码响应体无此字段 |
| `mismatch` | 需求与代码行为不一致 | 需求说库存不足返回 400，代码实际返回 200 带 error_code |
| `extra` | 代码有隐式限制，需求未提及 | 代码做了频率限制 10次/分，需求文档无此描述 |
| `risk` | 代码本身的潜在风险 | 并发场景下无锁操作，可能出现超卖 |

每条偏差输出：严重程度（high/medium/low）、需求原文、代码行为、**测试重点**、**修复建议**。

同时调用 `_generate_diff_cases()` 为每条偏差自动生成 1-2 条专项验证用例，前端汇总为差异验证用例表格，一键保存到项目用例库（`module="差异验证"`）。

**为什么不做成代码 Review 工具？**

代码 Review 告诉开发哪里写错了，目标是修代码。代码可行性分析告诉测试工程师哪里需要专项测试，目标是设计用例——视角和产物完全不同。两者面向不同受众，不混用。

**整体分析链路**：

```
输入：需求文档 + 接口代码 + 语言类型
    ↓
LLM 一次调用：对比两者，输出 risk_level + summary + items[]
    ↓
LLM 第二次调用：对 items[] 逐条生成差异验证用例 auto_cases[]
    ↓
前端展示：风险等级横幅 + 差异条目分组列表 + 验证用例表格
    ↓
一键保存：auto_cases 写入 ApiCase 表，module="差异验证"
```

## 七、变量池设计：打通执行链路

接口测试的核心难题是**用例间数据依赖**——登录接口拿到 token，后续接口必须带上这个 token。

平台设计了三层变量机制：

```
┌────────────────────────────────────────────────┐
│  全局变量池（global）                           │
│  跨项目、持久化到 DB                           │
│  引用方式：{{gvar:token}}                      │
├────────────────────────────────────────────────┤
│  本地变量（local）                              │
│  当次执行链内共享，内存级                       │
│  引用方式：{{var:user_id}}                     │
├────────────────────────────────────────────────┤
│  占位符函数                                    │
│  动态计算，执行时解析                          │
│  {{uuid()}}  {{timestamp()}}  {{fn(arg1,arg2)}}│
└────────────────────────────────────────────────┘
```

**测试计划中的关键设计**：所有步骤共享同一个 `var_store` 字典，前一步通过 JSONPath 提取的变量，后一步请求体或断言中可以直接引用，实现真正的**端到端链路测试**，而不是一堆孤立的接口调用。

**自定义脚本函数**的设计也值得一提：函数以 Python 代码存入数据库，执行时动态编译注入，用 `{{generate_req_sign(app_secret, uinfo, password)}}` 这种形式调用。测试人员只需填写参数，不接触 Python 运行时。

## 八、网络层可配置：面向真实测试环境

测试环境往往不是标准公网直连，常见问题：

- 内网服务需要走代理才能访问
- 测试域名需要解析到特定 IP（DNS 隔离）
- 不同环境用不同的网络出口

平台在 HTTP 执行层做了两层配置：

### 代理（Proxy）

```
支持格式：
  192.168.1.100:8080          ← 自动补 http://
  http://proxy.corp.com:3128
  socks5://user:pass@host:1080
```

### Hosts 映射

```
格式同 /etc/hosts：
  47.94.236.243 japi.hqwx.com
  192.168.1.10  api.dev.example.com
```

**实现原理**：自定义 `httpx.AsyncHTTPTransport` 子类，拦截 `handle_async_request`，将目标域名的连接重定向到指定 IP，同时保留原域名作为 `Host` 请求头，HTTPS 配合 `verify=False` 正常工作。

**为什么不直接改系统 hosts 文件？**

改系统 hosts 文件会影响本机所有进程，且需要管理员权限，每次切换环境都要手动改。平台的方案做到**应用内隔离**，不同项目可以有不同的映射规则，互不干扰。

**优先级链路**：

```
测试计划级 hosts > 项目级 hosts（同名条目计划覆盖，不冲突条目合并）
测试计划级代理  > 项目级代理
```

## 九、实时推送：全程 WebSocket，不轮询

所有长时间运行的任务（用例生成、增量更新、测试执行、压力测试、AI 分析）均通过 WebSocket 推送进度，前端**不做任何轮询**。

### 频道设计（按 client\_id 分组广播）

| client\_id  | 事件                     | 场景                  |
| ----------- | ---------------------- | ------------------- |
| `ai_gen`    | `ai_gen_progress`      | AI 用例生成/优化/增量更新进度   |
| `cases_gen` | `cases_gen_progress`   | UI 用例生成/增量更新进度      |
| `api_exec`  | `api_exec_progress`    | 接口用例执行结果            |
| `api_load`  | `load_metrics`         | 压测 TPS/P95/P99 实时指标 |
| `plan_<id>` | `plan_step_start/done` | 测试计划步骤进度            |

不同页面订阅不同 client\_id，互不干扰。

### 后端异步模型

```python
# HTTP 请求立即返回，执行在后台进行
background_tasks.add_task(_execute_plan_bg, plan_id)
return {"message": "已开始执行", "plan_id": plan_id}

# 后台任务通过 WS 推送每步结果
await ws_manager.broadcast_all({"type": "plan_step_done", ...})
```

**为什么不用轮询？**

压测场景下每秒推送一次 TPS 指标，轮询会产生大量无效请求；用例执行结果需要逐条实时展示，轮询的延迟和服务器负担都不可接受。WebSocket 是这类场景的唯一合理选择。

### 取消操作

生成/优化/增量更新过程中，前端通过 `AbortController` 中止 axios 请求，立即关闭弹窗，后端即使继续处理也不会将结果返回给前端：

```javascript
let _genAbortController = new AbortController()
await aiCaseApi.generate(payload, _genAbortController.signal)

// 用户点取消
_genAbortController.abort()
```

## 十、插件化技能架构：可扩展性

```
skills/
  ├── test_case_pro_max/      ← 放进去就自动注册
  │   ├── SKILL.md            ← 描述、配置、提示词
  │   └── scripts/run.py      ← 执行入口
  ├── page_parser/
  ├── document_parser/
  └── ...
```

**设计原则**：新增测试能力不修改主代码，扫描目录自动发现注册。

**为什么要插件化？**

测试场景千变万化，平台不可能内置所有能力。比如：

- 某项目需要特殊的签名算法 → 写一个自定义脚本注入
- 某团队想用增强提示词生成更规范的用例 → 新建技能目录
- 未来要支持 gRPC 测试 → 新建 grpc\_executor 技能

插件架构让平台的能力边界可以被业务方自己扩展，不依赖平台维护者。

## 十一、数据库设计：向前兼容，零停机升级

使用 SQLAlchemy ORM，启动时自动建表，新字段通过 `ALTER TABLE` 兼容旧库：

```python
for ddl in [
    "ALTER TABLE ai_case_files ADD COLUMN doc_hash VARCHAR(64)",
    "ALTER TABLE ai_case_files ADD COLUMN doc_content TEXT",
    "ALTER TABLE ai_case_files ADD COLUMN parent_id INTEGER",
    "ALTER TABLE ai_case_files ADD COLUMN diff_summary TEXT",
    "ALTER TABLE ai_case_files ADD COLUMN record_status VARCHAR(20) DEFAULT 'active'",
    "ALTER TABLE test_tasks ADD COLUMN doc_snapshot TEXT",
    "ALTER TABLE test_tasks ADD COLUMN doc_hash VARCHAR(64)",
    "ALTER TABLE test_cases ADD COLUMN deprecated BOOLEAN DEFAULT 0",
]:
    try:
        await conn.execute(text(ddl))
    except Exception:
        pass  # 列已存在则忽略（SQLite / PostgreSQL 均适用）
```

**双数据库支持**：

| 环境 | 数据库 | 驱动 | 适用场景 |
| --- | --- | --- | --- |
| 本机开发 | SQLite | aiosqlite | 零配置，1-5 人 |
| Docker 部署 | PostgreSQL 15 | asyncpg | 多并发，不限人数 |

两套环境只需修改 `DATABASE_URL`，SQLAlchemy 自动适配，代码无需任何改动：

```ini
# 本机 .env
DATABASE_URL=sqlite+aiosqlite:///./uitest_agent.db

# Docker .env.docker
DATABASE_URL=postgresql+asyncpg://uitest:uitest123456@db:5432/uitest
```

**为什么这样设计？**

面向个人/小团队的工具，不能要求用户在升级前先备份数据库、执行迁移脚本。\
这个方案的升级流程是：**直接拉代码 → 重启服务 → 旧数据库自动补列，历史数据完整保留**。

代价是不支持列的重命名和删除，但对于快速迭代的工具产品，这是合理的取舍。

## 十二、前端设计原则

### 统一的实时体验

所有耗时操作都有进度反馈，没有「点完按钮然后盯着空白页等」的场景：

- 进度弹窗 + 百分比（用例生成/优化/增量更新），支持随时取消
- 步骤列表实时滚动（测试计划执行）
- 实时折线图（压测指标）

### 需求变更的视觉反馈

增量更新后的用例按变更状态做视觉区分：

| 标记 | 显示 | 含义 |
|------|------|------|
| `is_new=True` | 绿色 NEW 标签 | 新需求新增的用例 |
| `is_updated=True` | 橙色「更新」标签 | 变更模块重生成的用例 |
| `status=deprecated` | 红色删除线 + 「废弃」标签 | 需求已删除的功能对应的用例 |
| 废弃模块 | 置灰 + 折叠 + 红色「已废弃」标签 | 整个模块被删除 |

废弃用例默认隐藏，通过「显示废弃用例」开关控制，不干扰日常查看。

### 数据展示层级化

报告不是一张大表格，而是分层展示：

- **卡片列表**：快速扫描整体通过率
- **详情抽屉**：展开看步骤级断言结果
- **AI 分析面板**：结构化解析，按标题/要点/正文分块渲染，支持复制

### 操作效率优化

- 链接式操作按钮（执行 | 编辑 | 删除）+ 分隔线，替代色块按钮，减少视觉噪声
- 批量操作工具栏（多选 + 批量删除/启用/禁用）
- 时间统一处理 UTC 偏移问题（后端存 UTC，前端补 `Z` 后缀转本地时区）
- 任务管理状态/环境列中文映射（`parsed` → 已解析、`test` → 测试环境等），原始值保留在数据层
- AI 用例生成模块内支持多选 + 批量删除，不同模块选中状态互相独立
- 右侧详情头两行布局：标题行在上，操作按钮按职责分组在下，按竖线分隔

### AI 分析报告展示优化

接口测试报告和测试计划报告的 AI 分析结果通过 `marked.js` 渲染 Markdown，做了两层优化：

**后端 prompt 格式约束**：system prompt 明确要求 `##` 一级标题、`-` 列表、禁用 `---` 分隔线，避免 DeepSeek 等模型输出杂乱格式。

**前端 CSS 精细化**：
- `h2` 加左侧蓝色竖线 + 渐变背景，章节层次清晰
- 列表符号统一蓝色，表格偶数行灰底
- 白底卡片替代灰底，视觉更干净

## 十三、页面元素抓取优化

WebUI 自动化依赖 Playwright 抓取页面可交互元素，早期使用 `wait_until="commit"`（只等 HTTP 响应头），动态渲染的 Vue/React 页面往往只抓到骨架屏的空 div，导致模块提取失败，用例质量极差。

**改造方案**：

```python
# 改造前
await page.goto(url, wait_until="commit")
await page.wait_for_timeout(5000)

# 改造后
# 1. 自动识别移动端 H5 页面（URL 含 m.、micropage、/h5 等），切换 iPhone UA
is_mobile = any(k in url for k in ['/m/', 'm.', 'micropage', 'h5'])
if is_mobile:
    await context.set_extra_http_headers({"User-Agent": "iPhone UA..."})

# 2. 等待网络请求静止，确保动态内容渲染完毕
await page.goto(url, wait_until="networkidle")

# 3. 元素过少时重试一次（二次兜底）
if len(elements) < 5:
    await page.wait_for_timeout(3000)
    elements = await page.evaluate(elements_script)
```

**为什么不直接用 `domcontentloaded`？**

`domcontentloaded` 只等 HTML 解析完，JS 还未执行；`load` 等静态资源加载完但 JS 渲染可能还未结束；`networkidle` 等所有网络请求静止，是最适合 SPA 应用的等待策略，代价是稍慢（通常多 1-2 秒），这对用例生成场景完全可接受。

## 十五、多用户并发设计

最初架构是单机单用户设计，`UITestAgent`、`BrowserPool`、`TestExecutor` 都持有全局共享状态，多用户同时操作会互相覆盖。经过方案一改造后，已支持团队局域网共享使用。

### 核心改造思路

**原问题：**
```
用户 A 执行 → uitest_agent.state.cases = A的用例
用户 B 执行 → uitest_agent.state.cases = B的用例  ← 覆盖了 A！
```

**改造后：**
```
用户 A 执行 → uitest_agent._states[task_id_A].cases = A的用例
用户 B 执行 → uitest_agent._states[task_id_B].cases = B的用例  ← 完全独立
```

### 三层隔离

**1. 浏览器 Context 隔离（`tools/browser.py`）**

```python
class BrowserPool:
    # 共享 Browser 实例（进程级，避免反复启动开销）
    # 每次请求创建独立 Context（完全隔离 cookie/storage/状态）
    # Semaphore(3) 限制最大并发，超出排队等待

    async def acquire(self, browser_type) -> BrowserTool:
        await self._semaphore.acquire()   # 排队等槽位
        browser = await self._ensure_browser(browser_type)
        bt = BrowserTool(browser, browser_type)
        await bt._init_context()          # 独立 Context
        return bt

    # 用完必须调用 release()，归还槽位
```

**2. Agent 状态隔离（`agent/core.py`）**

```python
class UITestAgent:
    def __init__(self):
        self._states: Dict[int, AgentState] = {}  # 按 task_id 隔离

    def _get_state(self, task_id: int) -> AgentState:
        if task_id not in self._states:
            self._states[task_id] = AgentState(task_id)
        return self._states[task_id]

    def _cleanup_state(self, task_id: int):
        self._states.pop(task_id, None)   # 任务结束自动清理
```

**3. 执行状态隔离（`skills/test_executor.py`）**

```python
class TestExecutor:
    def __init__(self):
        self._states: Dict[int, TaskExecutionState] = {}

    # 暂停/继续/停止精确控制单个任务，不影响其他用户
    def pause_task(self, task_id: int): ...
    def resume_task(self, task_id: int): ...
    def stop_task(self, task_id: int): ...
```

## 十六、一句话总结

> 平台的设计主线是**把 AI 能力嵌入测试生命周期的每个环节**：用插件化架构保证可扩展，用 WebSocket 保证实时体验，用需求变更增量更新避免旧用例丢失，用废弃/禁用字段分离保证执行精度，用可配置的网络层适配真实测试环境，用共享变量池打通接口链路，用统一的 HTTP API 适配任意大模型，用三层状态隔离支持团队并发使用，用接口代码直接驱动用例生成消除文档依赖，用代码可行性分析在代码阶段提前暴露需求与实现的偏差——整体目标是让测试工程师从重复劳动中解放出来，专注于测试策略和质量判断。
